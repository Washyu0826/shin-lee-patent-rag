## 一句話介紹

**把全台灣的發明專利，變成一個你可以「直接問問題」的 AI 助理 — 而且它會老實附上來源，不會亂掰。**

---

## 一、為什麼要做這件事？（動機）

### 真實情境：研發工程師的一天

小陳是台灣某家面板廠的研發工程師，主管丟給他一個任務：「我們想做一款『可降低眩光的偏光膜』，先去查有沒有人做過。」

小陳打開 TIPO（智慧財產局）的全文檢索系統。然後問題開始了：

| 工程師遇到的痛點 | 為什麼這是問題 |
|---|---|
| **只能用關鍵字 + AND/OR 布林查詢** | 他得自己想出「眩光」「防眩」「抗反射」「low glare」「anti-reflection」…然後一個個試，漏掉同義詞就漏掉專利 |
| **舊專利只有掃描 PDF** | 1980–2000 年的專利大量是掃描檔，**沒有可搜尋的文字**，等於是黑箱 |
| **跨領域查不到** | 「能源回收機構」可能藏在汽車、電梯、洗衣機三個完全不同的 IPC 分類底下 |
| **看不懂 claim 結構** | 專利申請範圍寫得像繞口令，看完一份 30 頁的專利要花一整天 |
| **問 ChatGPT 會被騙** | 通用 AI 對台灣專利沒有真實資料，會編出看起來很合理、但**根本不存在**的專利號 |

結果：小陳花了一整週才看完 50 件相關專利，主管又說「下週要看 200 件」。

### 我們的觀察

> **「找專利」不該像「考古」，應該像「跟一個讀過全部專利的助理對話」。**

這就是這個專案要做的事。

## 二、要做什麼？（目標）

### 三層目標

#### Level 1：可用（POC）
做出一個**可以 demo 的原型**，讓小陳可以：
- 上傳一份 PDF 專利 → 系統自動讀懂
- 用中文或英文自然語言問問題
- 收到答案 + **附上「這句話來自哪一篇專利的哪一段」**
- 點一下就能跳回原文 PDF 查證

#### Level 2：可信
- 答案**一定有引用**（系統設計強制要求附 [Source 1] [Source 2]）
- 引用是**真實的**（從向量資料庫拉出來的真實段落，不是 AI 編的）
- **本地運行**，資料不外送 — 公司內部研發資料不會跑去 OpenAI 的伺服器

#### Level 3：可量化
- 用「**SOTA 流水線 vs 普通方案**」的對照數字證明它真的有效
- 不是「我覺得不錯」，是「在這 4 種代表性問題上，準確度從 0.001 提升到 0.16，提升 160 倍」

## 三、需求與限制（為什麼長這樣）

### 業主的需求（昕力資訊）

| 需求 | 為什麼 | 我們怎麼做 |
|---|---|---|
| 中英文都要會 | 台灣專利 claims 是中文、IPC 分類碼是英文 | 用多語模型 (bge-m3) 同時處理 |
| 答案必須附引用 | 沒引用的 AI 回答在企業場景毫無價值 | 強制 prompt + 後處理檢查 |
| 要能看 PDF 原檔 | 法務 / 專利工程師最後一定要看原文 | UI 內建 PDF 預覽 + 高亮 |
| 部署要簡單 | 客戶可能要跑在自家機器上 | Docker Compose 一鍵啟動 |
| 不能漏資料 | 漏掉一件可能就違法 | 混合檢索 (向量 + 關鍵字 + 過濾器) |

### 硬體限制（POC 階段在筆電上跑）

- 筆電：**RTX 4060 Laptop GPU，VRAM 只有 8 GB**
- 不能用大型雲端 API（要錢、要連網、資料外送）
- 不能用 70B 級的大型 LLM（裝不下）
- → 所以選了 7B 的小模型 + 各種技巧把效果拉上來

### 法規 / 倫理限制

- 資料來源必須是**公開、合法**的 → 用 TIPO 政府開放資料 API
- 不能存使用者私資料 → 對話歷史只存在使用者瀏覽器（localStorage），不上雲
- 必須能完全離線 → 所有模型都本地跑

## 四、解決方案（白話設計）

### 系統怎麼運作？三步驟

```
①  灌專利進去             ②  使用者問問題             ③  AI 回答
   ┌──────────┐              ┌─────────┐              ┌──────────┐
   │ PDF/XML  │              │ "誰做過  │              │ "有兩家... │
   │ 1000 件  │ ───────────> │ 防眩偏  │ ─────────>   │ [Source 1]│
   │ 專利    │              │ 光膜?"  │              │ [Source 2]│
   └──────────┘              └─────────┘              └──────────┘
        │                          │                          │
        ▼                          ▼                          ▼
   切成小段、編碼、               用同樣方式             AI 一定要引用
   存進向量資料庫                 編碼問題、              不可以亂掰
                                 找最像的段落
```

### 為什麼比一般 AI 強？

#### 殺手鐧 1：「雙引擎」混合檢索

普通 AI 只用「**意思相近**」找答案 → 漏掉那些**用詞不同但意思一樣**的專利。

我們同時用兩種方式找：
- **意思引擎**（Dense）：「能源回收」→ 也能找到「動能再生」
- **字面引擎**（Sparse）：「IPC F02D41/00」→ 精準命中代碼

兩個結果合併排序，**漏掉的機率大幅降低**。

#### 殺手鐧 2：「三段式」品質過濾

```
你的問題 → [粗篩 30 件] → [Reranker 精排 Top 5] → [LLM 生成答案]
              快但粗         慢但準              帶引用
```

像是先用 Google 找 30 篇，再請一個專家從這 30 篇挑 5 篇精讀。

#### 殺手鐧 3：「假想答案」HyDE 技巧

當你問**模糊問題**（例如「有什麼節能的引擎？」），系統會先請 AI **腦補一段假想專利**，再用這段假想的去找真實專利。

聽起來怪，但效果是：**模糊問題的命中率從 0.001 提升到 0.16，160 倍**。原因是「假想專利」用的詞跟真實專利更像，比直接拿使用者口語去找好太多。

### 為什麼答案不會亂掰？

三道保險：

1. **AI 看不到自己沒有的東西** — 我們只把找到的真實段落餵給 AI，它答不出來只能說「資料中沒有」
2. **強制引用** — Prompt 寫死「每個事實都要附 [Source N]」，沒附的會被視為 fail
3. **可驗證** — 點 [Source 1] 直接跳到原文 PDF 那一頁，是真是假當場看得到

## 五、目前進度（2026-04-28）

### 已完成 ✅

- 完整 RAG 流水線（OCR → 切塊 → 雙引擎檢索 → 重排序 → LLM 生成）
- A/B 對照架構（baseline 普通方案 vs m3 SOTA 方案 並排比較）
- HyDE 技巧（模糊問題提升 160 倍）
- **真實 TIPO 資料 100 件**（涵蓋 IPC 八大分類 A/B/C/D/E/F/G/H）
- 完整前端 SPA（PDF 預覽、引用跳轉、雙語介面、回饋系統）
- 評測腳本（4 種代表性問題的自動化 benchmark）
- 本地完整離線運作（沒有任何雲端依賴）

### 數字證據（2026-04-28 在 100 件真實 TIPO 專利上跑出）

針對 4 種代表性問題類型，「Reranker top-1 信心分數」對照（0–1，越高越好）：

| 問題類型 | 普通方案 | SOTA 方案 (m3) | SOTA + HyDE |
|---|---|---|---|
| 直接事實題（IPC 列表） | 0.018 | 0.387 | **0.521** |
| 間接語意題（能源回收） | 0.0014 | 0.0014 | **0.889** ⭐⭐⭐ |
| 跨文件題（抗體） | 0.708 | 0.779 | **0.970** |
| 專業術語題（偏光膜） | 0.528 | 0.699 | **0.954** |

⭐ 最大亮點是**間接語意題**：普通方案幾乎搜不到（0.0014，相當於亂猜），m3+HyDE 跳到 **0.889 — 提升 635 倍**。這證明「使用者的口語提問」跟「專利的專業用詞」之間的語意鴻溝，靠單純向量化解決不了，必須加上 HyDE 這種「先請 AI 寫個假想答案再去找」的技巧。

### 系統反應速度（單次 query 端到端）

| Pipeline | 平均延遲 |
|---|---|
| 普通方案（baseline） | 16–21 秒 |
| SOTA 方案（m3） | 17–61 秒 |
| SOTA + HyDE（最強） | 34–41 秒 |
| 掃描 PDF OCR（4 頁） | 約 4.6 秒 |

HyDE 多花的時間用來換**質量提升**——在「間接語意題」這類使用者最常遇到的場景，是必要的代價。

## 七、為什麼這專案重要？

### 對學術
- 這是 **GDG on Campus** 第一個跟產業合作、有真實資料的 RAG 專案
- 技術棧（bge-m3, RRF, HyDE, Reranker）剛好是 2024–2025 業界 RAG 的 SOTA 組合，學完就是業界水準

### 對產業（昕力）
- 證明「**用本地小模型 + 好的檢索流水線**」可以打贏「**雲端大模型 + 普通檢索**」
- 為昕力既有產品線（digiRunner / Dify）提供一個有真實資料的 showcase
- POC 通過後可以推給法務 / 製造業客戶（這些客戶最在意「資料不外送」）

### 對使用者
- 把「找專利」從一週縮短到一杯咖啡
- 把「看不懂專利」變成「跟 AI 對話」
- 不用怕 AI 亂掰 — 每句話都有來源
