# Patent RAG Chatbot v3

> 一個會「翻書找答案」的 AI 助理 — 把專利檔案丟進來，用中文或英文問問題，它會找出相關段落、生成答案，**每句話都會告訴你來自哪一份檔案的哪一頁**。
>
> 學生作品 — 與 TPIsoftware（昕力資訊）合作的 GDG on Campus 專案。
> 在一台普通筆電（RTX 4060 顯卡、8 GB 顯卡記憶體）上完整運作。

---

## ⚠️ 這是什麼版本？

這是**概念驗證原型**（POC，proof of concept）。

預設帳密寫死成 `admin` / `patent2026`，是為了讓 demo 一鍵能跑。**任何要拿到網路上給多人用的場景，必須先改密碼**（在 `.env` 設定 `AUTH_PASSWORD` 與 `JWT_SECRET`）。

## 這個專案到底在解決什麼問題？

想像一個常見場景：

> *「我們公司去年累積了上百份專利、研發報告、合約。每次有人問『跟 OOO 相關的內容在哪幾份檔案？』，工程師要花一兩小時翻檔。」*

**現有的工具都不夠用：**

- **Ctrl+F 全文搜尋** — 你打「能源」搜不到「動能轉換」
- **直接問 ChatGPT** — 它根本沒看過你的內部檔案，要嘛複製貼上、要嘛它隨便編
- **商用搜尋平台** — 月費高、資料要上傳到別人雲端、無法客製

**這個專案做的事**：把所有檔案留在你**自己的機器**上，但仍然能用自然語言問問題，每個答案都會引用「這句話來自哪一份檔案的哪一頁」，可以點下去看原文 PDF 確認。

## 你跟它的對話會長這樣

```
你 ▸ 哪幾篇專利跟抗體分子有關？

AI ▸ 找到 2 篇相關專利：
     • I918591 — 含有替代凝血因子 VIII 功能之多特異性抗原結合分子
       [Source 1]
     • I918600 — PI4 激酶抑制劑及其使用方法 [Source 2]

     兩者都涉及生物分子的設計... [完整答案]

     [Source 1] [Source 2] ← 點下去會跳出對應 PDF 頁面
     ⏱ 25 秒 · 信心 HIGH ✓
```

```
你 ▸ 有沒有跟太空科技相關的專利？

AI ▸ ⚠️ 找到 3 個候選段落，但最高相似度只有 0.004
     （低於 0.05 的最低門檻）。我不會用低信心的內容硬答，
     請換個方式問或縮小範圍。

     信心：LOW
```

**關鍵差異**：當系統找不到夠相關的內容，它會**直接拒答**，不會像 ChatGPT 那樣編一個聽起來合理但錯誤的答案。

## 跟 ChatGPT 比的優勢

| 場景 | ChatGPT | 本專案 |
|---|---|---|
| 一般專利知識問答 | ✅ 很強 | ⚠️ 普通（用較小的本地模型）|
| **問「我們公司這 1600+ 個檔案」**| ❌ 沒看過你的資料 | ✅ |
| **答案自動引用回原檔** | ❌ | ✅ |
| **資料 100% 留在自己機器** | ❌（上傳到 OpenAI 雲端）| ✅ |
| **找不到時老實拒答** | ❌（常常硬編）| ✅ |
| **可看到「為什麼選這幾段」** | ❌ | ✅（Retrieval Funnel）|
| **批量問題的成本** | $$$ 隨用量計費 | 0（本機運算）|

**所以**：這不是「比 ChatGPT 厲害」 — 是**做 ChatGPT 不能做、做了也不能用的事**：私有資料、要可審計、要會拒答。

## 真實量化成果

跑了 100 篇真實台灣專利的對照實驗（資料來自 [TIPO 智慧財產局公開資料](https://cloud.tipo.gov.tw/S220/opdata/api/gazettes/P13)）。

### 不同問題類型，三種引擎的表現（信心分數 0–1，越高越好）

| 問題類型 | 傳統做法 | 改良版 | 改良版 + 加強 |
|---|---:|---:|---:|
| 直接事實（列出 IPC 分類） | 0.018 | 0.387 | **0.521** |
| **語意相關**（找間接關聯）| 0.0014 | 0.0014 | **0.889** |
| 跨檔案比較 | 0.708 | 0.779 | **0.970** |
| 專有名詞 | 0.528 | 0.699 | **0.954** |

**第二行**：問題的關鍵詞跟資料完全對不上時，傳統做法跟一般改良都失敗（0.0014），但加強後變成 0.889 — **提升 635 倍**。

> 這個「語意相關」測試題是**故意設計的硬題** — 100 篇語料中**沒有專利**用相同關鍵字。系統能找到語意間接相關的內容，這是這個架構真正解決的問題。

完整數據：[`data/eval/ab_bench_100patents.md`](data/eval/ab_bench_100patents.md)。

## 你能用它做什麼？

### 1. 問問題拿到帶引用的答案

進系統 → 拖一份 PDF/XML 到左上角 → 等系統處理（OCR + 索引，約 30 秒）→ 開始問問題。

### 2. 比對兩份檔案

選兩個專利編號 → 系統會並排顯示**書目資訊 / 摘要 / 第一個請求項**，可以一眼看差異。

### 3. 看「AI 為什麼選這幾段」

點右上角的「Retrieval funnel」會跳出 4 欄視窗，依序顯示：

1. **語意搜尋**找到的 top 候選 + 分數
2. **關鍵字搜尋**找到的 top 候選 + 分數
3. **兩者融合**後的排名
4. **重排序**（rerank）後的最終結果

這對學界 / 評審 / 主管來說很重要：可以**驗證系統不是黑盒子**。

### 4. 立刻換引擎重答比較

每個答案下面有 4 顆按鈕（Baseline / m3 / m3+HyDE / Auto），同一題可立刻換引擎重問，視覺對比差異。

## 馬上試（最簡單路線）

```bash
git clone https://github.com/Washyu0826/shin-lee-patent-rag.git
cd shin-lee-patent-rag
cp .env.example .env

# 一鍵拉起資料庫 + 安裝依賴 + 拉模型
make up && make pull && make install
make dev
```

打開瀏覽器 → http://localhost:8000/ → 自動登入。

**前置需求**：

- Python 3.11 以上
- Docker（資料庫跟向量庫用 Docker 起）
- [Ollama](https://ollama.com/)（本地 LLM 執行環境）
- NVIDIA GPU（建議；用 CPU 跑會慢很多）

完整安裝步驟見 [`docs/DEPLOYMENT.md`](docs/DEPLOYMENT.md)。

## 系統內部長這樣（簡略）

```
   你的瀏覽器
      ↓ 用問題打 API
   FastAPI 後端
      ├─→ Qdrant（記住所有 PDF 段落的「向量指紋」）
      ├─→ Ollama（本機跑的小型 LLM）
      └─→ PostgreSQL（記錄所有問答歷史，用於後續分析）
```

每個問題進來，系統會做 8 件事：

1. **判斷問題類型** — 是事實題、語意題、還是比較題？
2. **HyDE**（Hypothetical Document Embeddings）— 先讓小型 AI 寫一段「假設答案」當搜尋關鍵字
3. **雙路檢索** — 同時用語意 + 字面兩種方式搜尋
4. **融合排名** — 合併兩種搜尋結果
5. **重排序** — 用更精準的 AI 模型重新打分
6. **信心檢查** — 分數低於門檻直接拒答
7. **多樣性挑選** — 強制 top 5 來自不同檔案
8. **生成答案** — 串流產生，每段都帶引用

完整技術細節：[`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md)（有十幾節，包含完整 pipeline 圖、資料庫 schema、設計取捨）。

## 設計取捨（誠實寫出限制）

**強的部分**：
- 私有資料保留本機、答案會引用、找不到會拒答 — 是商用級的可審計設計
- 用一台普通筆電（RTX 4060 8GB）跑完整 pipeline，沒有任何雲端依賴
- 三種檢索引擎可以線上比較

**還沒做的（誠實標出）**：
- 沒接 SSO（單一登入）— 公司部署要再做
- LLM 是本地小模型 qwen2.5:7b — 對複雜推理不如 GPT-4
- 中文 ↔ 英文跨語言檢索仍依賴 m3 引擎，baseline 引擎不支援
- Kubernetes 設定還是 POC 草案，沒做負載平衡 / 私鑰輪換
- 對 ISO 標準、學校規章、司法文書等其他文件類型還沒寫對應的解析器
- PostgreSQL 沒設定的話，所有「歷史紀錄」會被靜默丟棄（沒 SQLite fallback）

**版權合規**：本機使用任何 PDF/XML 都 OK，但 commit 進 repo 要先確認授權，詳見 [`docs/DATA_SOURCES.md`](docs/DATA_SOURCES.md)。
